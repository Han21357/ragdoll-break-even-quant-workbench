# 面试讲解指南

## 推荐定位

不要把项目讲成“我做了一个量化网站”，而是讲成：

> 我观察到个人投资工具能够提供大量行情和观点，但很难保存一次判断的证据、约束与后验结果。因此我做了一套 evidence-first 投资研究工作台，把策略、持仓、AI 分析和复盘组织成可追溯闭环。

## 3 分钟版本

### 0:00-0:30 - 问题与判断

- 行情、策略、持仓和复盘通常分散。
- AI 很容易生成听起来合理、但无法核验的建议。
- 核心判断：专业感来自可解释、可制衡、可复盘，而不是更大胆的买卖结论。

### 0:30-1:30 - 走产品链路

1. 工作台：市场、组合、策略和复盘优先级汇总。
2. 策略研究：自然语言转 DSL，先确认规则和数据可用性。
3. 组合：模糊搜索导入持仓，保存交易流水和策略血缘。
4. AI 研究员：六角色独立取证，主席保留分歧。
5. 复盘：还原证据快照，区分判断偏差与执行偏差。

### 1:30-2:30 - 讲两项关键取舍

- 数据缺失不填 `0`：显示日期、来源、完整度和错误原因，必要时使用标记为 `stale` 的缓存。
- AI 不自动下单：用户确认后才写入正式决策，并同时保存失效条件和复盘日期。

### 2:30-3:00 - 给验证证据

- 可运行的 Flask/SQLite 源码，而不是只有原型。
- AKShare 主源与多备源降级。
- 31 项测试覆盖数据回退、策略 DSL、持仓冲突、交易流水和复盘。
- 公网只提供脱敏静态演示；动态后端默认仅绑定 `127.0.0.1`。

## 适合深挖的五个问题

1. **为什么不用一个大模型直接回答？**
   - 因为事实检索、风险检查、反方论证和最终决策承担不同责任，需要独立状态、证据和失败处理。
2. **多数据源如何避免字段混乱？**
   - DataProvider 统一字段、日期、来源、完整度和缺失原因，页面只消费内部契约。
3. **如何评价 AI 是否有用？**
   - 不只看收益，还记录证据覆盖、用户修改、判断正确性、执行正确性和偏差类型。
4. **为什么使用 SQLite？**
   - 当前是个人本地工作台，SQLite 便于审计与迁移；多人生产化时再升级受管数据库和权限体系。
5. **最大的产品风险是什么？**
   - 用户把研究意见误解为确定性建议，所以必须强调数据缺口、人工门禁、失效条件和非自动交易边界。

## 面试材料使用顺序

1. 先发 GitHub README 或项目 PDF。
2. 现场打开静态交互演示，走一条完整链路。
3. 被追问实现时再打开架构、DataProvider、测试和部署安全文档。
4. 不在公司设备上通过任何临时隧道暴露动态服务。
